# プロダクト要求定義書 (Product Requirements Document)

**バージョン**: 1.0
**作成日**: 2026-03-20
**プロジェクト**: FaceGraph

---

## プロダクト概要

### 名称
**FaceGraph** — 顔類似度定量分析アプリケーション

### プロダクトコンセプト
- **幾何学的特徴量による客観的スコア化**: 「なんとなく似ている」という主観を排し、顔のランドマーク座標から算出した数値（角度・距離・比率）で類似度を0〜100%のスコアとして出力する
- **アップロードだけで自動解析**: 画像アップロードをトリガーに、ランドマーク検出・特徴量抽出・統合を自動でバックグラウンド実行。比較リクエスト時は計算済みデータをそのまま使用する
- **ローカル完結・シングルユーザー**: Mac上のDockerコンテナで完全動作。認証不要。外部サービス不使用でプライバシーを保護

### プロダクトビジョン
子供が両親のどちらに似ているか、あるいは任意の2人がどれだけ顔が似ているかを、数値として客観的に提示するツールを提供する。ユーザーは正面顔画像をアップロードするだけで、顔パーツの幾何学的特徴量に基づく類似度スコアを得られる。ローカルDockerで動作するため、プライベートな顔画像を外部に送信せずに安心して利用できる。MediaPipeによる高精度ランドマーク検出（468点）と、拡張性を考慮したStrategyパターンの設計により、将来的な手法追加にも柔軟に対応できる。

### 目的
- 「どちらに似ているか」の感覚的な議論を定量スコアで補完する
- MediaPipe Face Meshを活用したGPU不要・高速な顔解析パイプラインを実現する
- 解析結果をDBにキャッシュし、過去の比較を即座に再表示できるようにする
- Strategyパターンにより、ランドマーク検出・特徴量抽出・類似度計算の手法を将来差し替え可能にする

## 成功指標(KPI)

### プライマリーKPI（MVP完了基準）
- `docker compose up` で4コンテナが起動し、ブラウザから操作できること（セットアップ成功率100%）
- 正面顔画像アップロード後、30秒以内に解析が完了し `analyzed` ステータスになること
- 2人の人物を選択してコサイン類似度スコア（0〜100%）が1回表示されること
- 顔が0個・2個以上の画像でエラーが返ること（バリデーション精度100%）
- 同じペアの再比較がキャッシュから返ること（再計算なし）

### セカンダリーKPI
- 画像追加・削除後に関連する比較結果が自動無効化されること
- 比較履歴一覧で過去の有効/無効ステータスを確認できること
- モデルバージョン管理により、手法変更時に特徴量の再計算が正しく行われること

## 機能要件

### コア機能（MVP）

#### 1. 人物管理

**ユーザーストーリー**:
家族写真を整理したいユーザーとして、誰の顔画像かを管理するために、人物を名前で登録・一覧・削除できる機能が欲しい

**受け入れ条件**:
- [ ] 名前（最大100文字）を入力して人物を作成できる
- [ ] 人物一覧にサムネイルと画像枚数が表示される
- [ ] 人物を削除すると、関連する画像・特徴量・比較結果が連鎖削除される

**優先度**: P0（必須）

---

#### 2. 画像登録・バリデーション

**ユーザーストーリー**:
類似度を計算したいユーザーとして、解析に使える正面顔画像を複数枚登録するために、アップロードと顔数チェックが自動で行われる機能が欲しい

**受け入れ条件**:
- [ ] 1人物につき複数画像をアップロードできる
- [ ] 顔が0個の画像：エラー「顔が検出されませんでした」を返す
- [ ] 顔が2個以上の画像：エラー「複数の顔が検出されました。1人だけ写っている画像を使用してください」を返す
- [ ] 顔が1個の画像：`analyzing` ステータスでレスポンスを返し、バックグラウンド解析を開始する
- [ ] 画像はMinIO（S3互換）に保存される

**優先度**: P0（必須）

---

#### 3. 顔解析パイプライン（バックグラウンド自動実行）

**ユーザーストーリー**:
ユーザーとして、アップロード後に手動操作なしで解析が完了するために、バックグラウンドで自動的にランドマーク検出・特徴量抽出が実行される機能が欲しい

**受け入れ条件**:
- [ ] 画像アップロード後、FastAPIのBackgroundTasksで自動的に解析パイプラインが起動する
- [ ] ステータス遷移: `uploaded → validating → analyzing → analyzed`（エラー時は `error`）
- [ ] MediaPipe Face Meshで468点のランドマーク座標を検出する
- [ ] IPD正規化距離比率ベクトルを算出し `features` テーブルに保存する
- [ ] 解析完了後、`person_features`（複数画像の平均統合）を更新する
- [ ] 解析完了後、その人物を含む `comparisons` を `is_valid = false` に更新する

**ステータス遷移**:
```
uploaded → validating → analyzing → analyzed
                  ↘ error（顔0個 / 複数顔 / 解析失敗）
```

**優先度**: P0（必須）

---

#### 4. 特徴量の定義

**ユーザーストーリー**:
開発者として、顔の幾何学的特徴を再現性高く比較するために、IPD正規化された距離・角度・比率ベクトルが算出される機能が欲しい

**受け入れ条件**:
- [ ] **距離特徴量**: 両目間距離（IPD、正規化基準）、鼻の長さ、口幅、顔幅、眉間距離、目と口の垂直距離をIPDで正規化して算出する
- [ ] **角度特徴量**: 目頭-鼻尖-目頭、口角-鼻尖-口角、眉端-目頭-鼻根の角度を算出する
- [ ] **比率特徴量**: 顔の縦横比、目幅/顔幅、鼻の長さ/顔の高さ、口幅/IPD、三分割比率（上顔/中顔/下顔）を算出する
- [ ] 特徴量ベクトルは `features.raw_vector` に FLOAT[] として保存される

**優先度**: P0（必須）

---

#### 5. 複数画像の統合

**ユーザーストーリー**:
複数の写真を登録したユーザーとして、より安定した類似度スコアを得るために、複数画像の特徴量が自動的に平均化される機能が欲しい

**受け入れ条件**:
- [ ] 1人物の全画像の特徴量ベクトルを平均して `person_features.feature_vector` に保存する
- [ ] 画像の追加・削除のたびに `person_features` を再統合する
- [ ] `person_features.image_count` に統合に使用した画像数を記録する

**優先度**: P0（必須）

---

#### 6. 類似度計算・比較

**ユーザーストーリー**:
2人の類似度を知りたいユーザーとして、比較結果を数値で確認するために、2人の人物を選択してコサイン類似度スコアを表示できる機能が欲しい

**受け入れ条件**:
- [ ] 任意の2人物を選択してコサイン類似度スコア（0〜100%）を表示できる
- [ ] 比較結果はDBにキャッシュし、`is_valid = true` の場合は再計算せずに返す
- [ ] `is_valid = false` または未計算の場合は自動的に再計算する
- [ ] レスポンスに `{ score: 0.72, is_cached: true }` の形式でキャッシュ判別が含まれる
- [ ] 同ペア・同手法での UNIQUE 制約により重複比較結果を防ぐ

**優先度**: P0（必須）

---

#### 7. 比較履歴

**ユーザーストーリー**:
過去の比較を振り返りたいユーザーとして、いつどの2人を比較したかを確認するために、比較結果の履歴一覧を表示できる機能が欲しい

**受け入れ条件**:
- [ ] 過去の全比較結果が一覧表示される
- [ ] 各比較に有効（`is_valid = true`）/ 無効（`is_valid = false`）のステータスが表示される

**優先度**: P0（必須）

---

### Post-MVP 機能

#### 人物の編集（名前変更）
削除→再作成で代替可能なため、Phase 2 で対応。
**優先度**: P1

#### メトリクス切り替えUI（ユークリッド距離等）
Strategyパターンで実装済みの手法をUIから選択できるようにする。
**優先度**: P1

#### 統合方法切り替えUI（中央値・単一選択）
平均方式以外の統合方法をUIから選択できるようにする。
**優先度**: P1

#### ジョブキュー（Celery等）
BackgroundTasksで十分なためPhase 2。大量処理や障害対応が必要になった場合に追加。
**優先度**: P1

#### パーツ別内訳表示
スコア全体ではなく、目・鼻・口など部位別の類似度を表示する。
**優先度**: P2

#### Procrustes解析
ランドマーク点群を最適重ね合わせ後の残差で類似度を算出する。
**優先度**: P2

#### 3人以上の一括比較
複数人を選択して全ペアを一括計算し、マトリクス表示する。
**優先度**: P2

## 非機能要件

### パフォーマンス
- 画像アップロードのバリデーションレスポンス（顔数チェック）: 5秒以内
- 1枚の画像の解析パイプライン完了: 30秒以内（Mac CPU環境）
- 比較結果のキャッシュ返却: 500ms以内
- ブラウザでの画面表示: 2秒以内（ローカル環境）

### ユーザビリティ
- `docker compose up` から初回ブラウザアクセスまで10分以内に完了できる
- 画像アップロードから比較スコア表示まで、UIのみで操作が完結する
- エラー発生時は具体的なエラーメッセージを日本語で表示する

### 信頼性
- 解析パイプラインのエラー（顔検出失敗等）は `images.metadata` にエラー理由を保存し、ステータスを `error` にする
- 画像変更時の比較結果無効化は整合性を保証する（トランザクション管理）
- モデルバージョン管理（`features.model_version`）により、手法変更時の特徴量の一貫性を保証する

### セキュリティ
- 認証不要・ローカル専用（`localhost` のみからのアクセスを想定）
- 顔画像は外部サービスへ送信しない（すべてローカル処理）
- MinIOのアクセスキーは環境変数で管理し、コードにハードコードしない

### スケーラビリティ（MVP時点の想定範囲）
- 登録人物数: 最大100人
- 1人物あたりの画像数: 最大20枚
- 比較結果キャッシュ: 数千件まで対応
- 同時接続: シングルユーザー前提（同時1接続）

## APIエンドポイント一覧

**人物管理**

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/persons` | 人物作成 `{ name: string }` |
| GET | `/api/persons` | 人物一覧（サムネイル・画像枚数付き） |
| GET | `/api/persons/{id}` | 人物詳細 |
| DELETE | `/api/persons/{id}` | 人物削除（画像・特徴量・比較結果を連鎖削除） |

**画像管理**

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/persons/{id}/images` | 画像アップロード（`multipart/form-data`）|
| GET | `/api/persons/{id}/images` | 画像一覧（ステータス付き） |
| DELETE | `/api/images/{id}` | 画像削除 → 特徴量再統合 → 比較無効化 |

**比較**

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/comparisons` | 比較実行 `{ person_a_id, person_b_id }` |
| GET | `/api/comparisons` | 比較結果履歴一覧 |
